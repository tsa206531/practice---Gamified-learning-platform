# 專案技術評估與建議（V411_17RE）

狀態：根據工作區前端目錄的關鍵檔案與結構進行快速審視，未對程式碼做任何修改。

## 總體評價
- 整體方向正確：使用 Next.js App Router、TypeScript、Tailwind v4、shadcn/ui 與 Radix 組件，能快速搭建一致且可維護的 UI。
- 專案定位清晰：學習平台導向（繁體中文、遊戲化元素、課程/排行/挑戰等版位已規劃）。
- 目前偏前端 MVP：backend 資料夾尚未落實，資料流/權限/API 尚未建立。建議盡快定義端到端 Domain 邏輯與資料契約。

## 已觀察到的優點
- 組件化與可重用性佳：`MainLayout`、`SidebarNav`、`Header`、UI atoms 抽象適當，已使用 `cn`、CVA variants 等常見模式。
- 中文在地化初步到位：標題、文案採用 zh-TW，利於後續 i18n 擴展。
- 專案結構清晰：Next app 分頁架構乾淨，UI 元件集中於 `components/ui`，可持續演進成設計系統。

## 風險與技術債
- 風險設定
  - `next.config.mjs`: `typescript.ignoreBuildErrors: true` 可能遮蔽型別問題，長期降低信心。
  - `images.unoptimized: true` 停用 Next Image 最佳化，可能影響 Lighthouse/CLS/流量成本。
  - `tsconfig.json`: `allowJs: true` 在 TS 專案不必要時可移除以拉高嚴謹度。
- 版本相依
  - React 19、Next 16、Tailwind 4：屬於較新版本，需留意三方套件相容與型別支援。
- 架構未落實
  - 無狀態管理策略（Server Components + 客戶端互動界線、全域狀態、快取策略未定）。
  - 無 API 合約/Domain 模型（課程、單元、挑戰、徽章、用戶/等級/XP）。
  - 無驗證/授權（UI 有登入登出按鈕，但未見 Auth 流程與 Token 管理）。
- 工程化缺口
  - 無 E2E/單元測試配置（Jest/RTL/Playwright 未見）。
  - 無 CI/CD、Linter/Formatter/Commit 規範（ESLint script 有但未見配置檔、Prettier 未見）。
  - 無環境變數管理策略（dotenv、schema 驗證未見）。
  - 無 Storybook/設計系統文檔。

## 建議的目標架構（前後端整合）
### 前端
- Next.js App Router：以 Server Components 為預設；互動區塊用 Client Components。
- 資料讀取：`fetch` + `revalidate`（RSC），或在 CSR 場景使用 React Query/SWR。
- 型別與資料契約：以 Zod 定義 DTO schema，前後端共享 types（`/types` 或 `/packages/shared`）。
- 狀態管理：優先使用伺服器快取與 URL state；必要時再引入 React Query 或 Zustand。
- i18n：`next-intl` 或 `next-i18next`，保留多語擴展能力。
- UI/設計系統：持續以 shadcn/ui + Tailwind tokens；導入 Storybook 做可視化文檔。

### 後端
- 選項 A：Next Route Handlers（API routes）+ Edge/Node runtime，快速整合部署。
- 選項 B：獨立 BFF（NestJS/Express/Fastify），清晰 Domain，便於對外整合與規模擴展。
- 可觀測性：日誌、追蹤（OpenTelemetry）、錯誤告警（Sentry）與 metrics 基礎設計。
- 資料庫：Prisma + Postgres（或 Supabase）；規劃權限/多租戶/審計欄位。

### Auth
- NextAuth/Auth.js + OAuth/Email/Passkey；或自研 Token（JWT + session cookie）。
- RBAC/ABAC 規則以中介層封裝，前端以能力標記控制 UI。
